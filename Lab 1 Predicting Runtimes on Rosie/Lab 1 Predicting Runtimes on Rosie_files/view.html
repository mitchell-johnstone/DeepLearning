Introduction
In this week’s experiment, we aim to get a better of understanding of how to estimate the time required to train a neural network and
how it relates to the number of CPUs, GPUs, and the type of GPU, available to our task. Through our experiment we also aim to
become more familiar with working in a Linux-based HPC environment. This includes becoming more comfortable navigating the
command line, requesting nodes via srun , and running singularity containers.
Experiment
Predicting Relative Runtime for the V100 vs. the T4 for Training a Neural Network
If we are assuming that the neural network is trained primarily with the GPU, we expect the GPU performance to be the main
bottleneck / determining of speed. So we can get an idea of the relative training times using the relative GPU performance.
FLOs, are one of the primary bottlenecks of a processing unit because floating point operations are much more costly compared to
integer arithmetic. Therefore we can use FLOs per second, or FLOPS, as a good heuristic of the performance of a processing unit.
Based on the ratio of TFLOPS for the V100 and the T4, TFLOPS, we expect the V100 to run approximately the number of
floating point operations, or FLOs, in the same time frame as the T4.
Knowing that FLOPS are a good estimate of program run time, especially for neural network training which involves heavy usage of
FLOs, we can guess that because the V100 can accomplish twice the number of FLOs in the same time frame as the T4, it will train
approximately twice as fast.

Discussion
While our predictions were not fully accurate, we found that we were no more than ~17% off from the actual runtime observed with
20 epochs. Our heuristic was that the time to train for 20 epochs should be the time for 1 epoch. This fails to account for
differences in performance due to OS scheduling and other jobs being started on the cluster which impacts the training time, which
is most likely the cause of the discrepancy.
In particular for the V100 and the T4, we were expecting to see the V100 run approximately twice as fast as the T4 based on their
respective TFLOPS. For the 1 epoch time value we saw that the V100 ran times faster than the T4 which is in
line with what we expect.

Lab 1: Predicting Runtimes2For the 20 epoch trial we saw that the V100 ran times faster. While the value is still within the realm of
predictions, we see that the V100 is starting to approach being ~3 times faster than the T4. We can guess that the cause of this drift
is due to OS scheduling & other jobs being started / ended on the node throughout training, much like the probable cause of the
error in our other predictions. There are also architectural differences between the two GPUs which may account for some of the
differences. As mentioned, while TFLOPS are good heuristics for comparing performance because FLOs are usually the bottleneck
for a device, not all operations run through the GPU may be FLOs. For example, there are also memory load/stores. The V100 and
the T4 also have different memory bandwidths (900 GB/s and 320 GB/s respectively), which is the rate at which they read data in
and out of memory. Since the V100 has a much higher memory bandwidth, it will pull ahead of the T4 when it comes to memory
operations. In one epoch there are probably not enough load/stores to make a difference, but over the course of 20 epochs it is
feasible that there is enough usage of memory to make a more noticeable difference.
To achieve more accurate results what we should do is repeat our time trials several times, ideally on a node with very low activity to
avoid other tasks on the node impacting performance.
